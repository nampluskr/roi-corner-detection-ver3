# project-start-guide.md

이 문서는 새 프로젝트를 문서기반 단계별 운영 방식으로 시작할 때, 첫 세션에서 참고하는 범용 가이드다. 특정 도메인이나 기술 스택에 종속되지 않으며, 어떤 신규 프로젝트에도 동일하게 적용한다.

## 1. 목적

이 가이드는 프로젝트를 코드부터 작성하지 않고, **문서를 먼저 확정한 뒤 그 문서를 기준으로 구현**하는 운영 방식을 새 프로젝트에 부트스트랩하기 위한 것이다.

이 방식의 핵심 개념은 두 가지다.

- canonical 문서 우선: 요구사항이나 설계가 바뀌면 코드보다 canonical 문서를 먼저 수정한다. 문서가 항상 단일 기준(single source of truth) 역할을 한다.
- plan 기반 실행: 모든 실제 작업은 `docs/plans/`의 계획 문서에 범위와 완료 기준을 기록하고, 검토·승인을 받은 뒤 실행한다.

이 가이드를 따르면 첫 세션 종료 시점에 프로젝트가 자체적으로 이 워크플로우를 갖추게 된다.

## 2. 전제와 적용 범위

이 가이드는 프로젝트 *내부*의 문서 체계를 부트스트랩하는 절차만 다룬다. 워크스페이스 전체의 상태 폴더 관리 규칙(`_core/rules/`에 정의됨)과는 별개다.

적용 전제는 다음과 같다.

- 워크스페이스 규칙상 새 프로젝트는 먼저 `00_review/` 상태 폴더 아래에 등록된다.
- 프로젝트 폴더명은 소문자와 하이픈 구성을 권장한다.
- 이 가이드가 만드는 문서는 모두 해당 프로젝트 폴더 *안*에 위치하며, 워크스페이스 운영 문서(`_core/`)를 수정하지 않는다.

## 3. 첫 세션에서 만들 산출물

첫 세션의 목표는 아래 문서와 폴더를 프로젝트 루트에 갖추는 것이다. 각 산출물의 책임과 대상 독자는 다음과 같다.

| 산출물 | 책임 | 대상 독자 |
| --- | --- | --- |
| `README.md` | 제품 소개 (현재 상태, 주요 기능, 개발 환경, 배포 계획) | 프로젝트를 처음 접하는 독자 |
| `AGENTS.md` | 운영 규칙 (문서 작성 규칙, 변경 작업 순서, plan 규칙, 변경 관리 규칙) | 작업 수행자 (사람 또는 에이전트) |
| `CLAUDE.md` | Claude Code 전용 진입점. 공통 규칙은 `AGENTS.md`에 위임하고, Claude 전용 규칙만 추가 | Claude Code |
| `docs/README.md` | 기술 문서 색인 + 문서 작성 워크플로우의 canonical 정의 | 상세 기준을 찾는 개발자 |
| `docs/plans/` | 계획 문서와 그 이력을 보관하는 폴더 | 작업 수행자 |

`CLAUDE.md`는 공통 규칙을 중복 서술하지 않고 `AGENTS.md`로 위임하는 최소 문서로 둔다. Claude 전용 규칙이 실제로 필요해질 때만 해당 내용을 추가한다.

## 4. 문서 간 책임 분리 원칙

각 문서는 하나의 책임만 지고, 동일한 규칙을 여러 문서에 중복 정의하지 않는다. 중복은 문서 간 불일치의 원인이 되므로 금지한다.

책임 분리의 기본 원칙은 다음과 같다.

- 제품이 무엇인지는 `README.md`에서만 서술한다.
- 작업을 어떻게 수행하는지(운영 규칙)는 `AGENTS.md`에서만 정의한다.
- 세부 기준(명명 규칙, plan 번호 규칙 등)은 `docs/README.md`에서만 canonical하게 정의하고, `AGENTS.md`는 그 절을 참조만 한다.
- 구체적 작업 범위와 완료 기준은 각 plan 문서에서만 기록한다.

한 규칙을 어느 문서에 둘지 애매하면, canonical 정의는 `docs/README.md`에 두고 나머지 문서는 링크로 참조하는 쪽을 택한다.

## 5. 초기 폴더 구조

초기에는 실제로 필요한 폴더만 만든다. 빈 폴더를 미리 남발하지 않고, 새 폴더는 그 필요를 정당화하는 승인된 plan과 함께 추가한다.

첫 세션 종료 시점의 최소 구조는 다음과 같다.

```text
{project}/
├── README.md
├── AGENTS.md
├── CLAUDE.md
└── docs/
    ├── README.md
    └── plans/
        └── 0001-....-plan.md
```

`docs/specs/`(제품·API 계약), `docs/design/`(구조 설계) 등의 하위 폴더는 필요해지는 시점에 후속 plan으로 추가한다. 초기 범위에서 이런 폴더를 미리 만들지 않는다.

## 6. plan 문서 규칙

모든 실제 작업은 계획 문서에 기록한다. 계획 문서의 경로와 명명 규칙은 다음과 같다.

- 경로: `docs/plans/NNNN-topic-plan.md`
- 번호 `NNNN`: 4자리 0-padding, 순증가. 번호를 재사용하거나 삭제하지 않는다.
- 상태: `Draft`, `Approved`, `Done` 중 하나를 사용한다.
- 완료된 plan도 파일을 지우지 않고 이력으로 보존한다.

각 plan 문서는 다음 구성 요소를 갖춘다.

- 표준 헤더 표: 상태, 작성일, 적용 범위, 관련 문서
- 목적과 배경: 이 작업이 왜 필요한지
- 범위: 포함 항목과 제외 항목을 구분해서 기록
- 완료 기준: 무엇이 충족되면 이 plan을 `Done`으로 볼 수 있는지

## 7. 첫 세션 진행 순서

첫 세션에서 진행하는 표준 순서는 다음과 같다. 핵심은 **첫 plan(`0001`)이 루트 문서와 `docs/README.md` 뼈대를 채우는 작업 그 자체**가 되어, 문서 워크플로우를 스스로 부트스트랩한다는 점이다.

1. 프로젝트의 성격과 현재 코드 상태를 파악한다. 기존 코드나 참고 자료가 있으면 실제 상태를 확인한다.
2. `docs/plans/0001-<topic>-plan.md` 초안을 작성한다. 이 첫 plan의 작업 대상은 루트 `README.md`/`AGENTS.md`/`CLAUDE.md`와 `docs/README.md`의 본문 작성이다.
3. 사용자에게 초안을 제시하고 검토·승인을 받는다.
4. 승인된 `0001` plan에 따라 루트 문서 3개와 `docs/README.md`를 실제로 작성한다.
5. 이후 모든 작업(추가 문서, 설계, 구현)은 새 plan(`0002`부터)으로 이어간다.

실제 사례로, 이 워크스페이스의 `00_review/260717_fft-analysis-gui-ver0` 프로젝트는 `0001`에서 루트·docs 뼈대를 세우고, `0002`에서 상세 문서 체계를 수립하는 방식으로 이 순서를 그대로 따랐다.

## 8. plan 파일 템플릿

첫 plan을 작성할 때 아래 골격을 복사해서 사용한다. 도메인에 맞게 제목과 각 절의 내용을 채운다.

```markdown
# <작업 제목>

| 항목 | 값 |
| --- | --- |
| 상태 | Draft |
| 작성일 | YYYY-MM-DD |
| 적용 범위 | <이 plan이 다루는 파일·영역> |
| 관련 문서 | <링크> |

## 1. 목적과 배경

<이 작업이 왜 필요한지, 현재 상태가 어떤지>

## 2. 범위

포함:
- <포함 항목>

제외 (후속 plan에서 수행):
- <제외 항목>

## 3. 완료 기준

- <무엇이 충족되면 Done인지>

## 4. 검증

<빌드/테스트 방법, 또는 문서 생성만이면 그 사실과 확인 항목>
```

## 9. 참고 자료

이 가이드가 일반화한 실제 사례와 문서 스타일 규칙은 다음에서 확인한다.

- `00_review/260717_fft-analysis-gui-ver0/AGENTS.md` — 운영 규칙 문서의 실제 예시
- `00_review/260717_fft-analysis-gui-ver0/README.md` — 제품 소개 문서의 실제 예시
- `00_review/260717_fft-analysis-gui-ver0/docs/README.md` — 문서 색인과 워크플로우 canonical 정의의 실제 예시
- `00_review/260717_fft-analysis-gui-ver0/docs/plans/0001-root-and-docs-index-plan.md` — 부트스트랩 첫 plan의 실제 예시
- `_core/rules/docs-rules.md` — 모든 마크다운 문서에 적용되는 문서 작성 스타일 규칙
